// 01What Is Function Calling — Really?

Most explanations of function calling start with a JSON schema. That's the wrong place to start. Start with what the model actually cannot do.

An LLM cannot check the weather. It cannot query your database. It cannot call your portfolio API. All it can do is generate text — and within that text, it can describe, in structured form, a request for something it needs. The application layer reads that request, decides whether to honour it, executes the actual function, and returns the result. The model folds that result into its next response.

That loop — model requests, application executes, model continues — is function calling. Everything else is engineering around it.

The model is not calling functions. It is emitting structured text that describes what it wants called. Your application decides what happens next.

Here's a concrete example. When you ask Claude "what's the weather in New York?" — Claude doesn't know the weather. Internally, it emits something like:

{
  "type": "tool_use",
  "name": "get_weather",
  "input": {
    "location": "New York",
    "units": "fahrenheit"
  }
}

Your application layer intercepts that, calls the actual weather API, gets the result, and passes it back. The model then writes the response the user sees. The user sees none of this. But a full request-response loop just completed — and that loop is the fundamental primitive of everything that follows.

// 02The Financial Planning Architecture

Let's make this concrete. A user asks: "Am I on track to retire at 62?"

That single question triggers a chain that exposes every hard problem in function calling simultaneously. Here's what actually happens:

Figure 2 — Multi-agent financial planning architecture. One user question cascades through orchestrator → specialist agents → tool layer → deterministic code.

Notice what's happening at each layer. The LLM orchestrator handles language understanding and routing — this is where probabilistic reasoning belongs. Specialist agents handle domain-specific coordination. The tool layer wraps deterministic functions. The deterministic code layer is pure math: same input, same output, every time.

Each hop in this chain is a potential failure point. What if the tax service times out on call three? What if the Monte Carlo model returns a probability range and the model doesn't know how to reason about uncertainty? What if a user input redirects which account gets queried?

// 03The Deadlock Incident

Six months ago, building this system, we experienced every engineer's nightmare: an infinite loop that burned our entire token budget before we knew what happened.

The architecture had an orchestrator agent and three specialist agents. We made one mistake: we trusted the orchestrator completely. Our prompt said, roughly: "you are a financial planning orchestrator, here are your agents, decide who to call."

What we didn't account for: when the retirement agent received an ambiguous request, it tried to gather more context — by calling back up to the orchestrator. The orchestrator, not recognising this as a re-entry, treated it as a new request and called the retirement agent again.

Figure 3 — The deadlock: Orchestrator calls Retirement Agent; Retirement Agent calls Orchestrator; repeat until token budget is exhausted

What We Fixed

Three independent defences, not one:

# 1. Hard token budget per agent (middleware, not prompt)
class AgentBudgetMiddleware:
    MAX_TOKENS_PER_AGENT = 4000

    def before_invoke(self, session_id, agent_name, token_count):
        running = self.tracker.get(session_id, agent_name)
        if running > self.MAX_TOKENS_PER_AGENT * 0.9:
            raise GracefulBudgetExit(
                agent=agent_name,
                message="Budget ceiling reached — returning partial result"
            )

# 2. Call depth counter (catches loops before exhaustion)
def check_call_depth(session_id, window_seconds=60, max_calls=12):
    recent = call_log.count_since(session_id, window_seconds)
    if recent > max_calls:
        raise LoopDetected(f"Session {session_id}: {recent} calls in {window_seconds}s")

# 3. Orchestrator prompt now forbids upward delegation
ORCHESTRATOR_SYSTEM = """
You are a financial planning orchestrator.
STRICT RULES:
- You ONLY call tools. You do NOT call other agents.
- If a specialist agent calls you, REJECT the request and log a warning.
- If you cannot route with confidence, return a clarification request to the user.
- Never attempt to resolve ambiguity by calling multiple agents sequentially.
"""
You cannot rely on the model to know when to stop. The circuit breaker must live in your code, not your prompt.

// 04The Hybrid Deterministic Architecture

We ran an experiment early on: ask the LLM to compute a retirement projection directly from the prompt. Same inputs, five runs. We got four different answers. The differences were small — but in financial services, small differences in projected returns compound into large differences in advice. We would never ship that.

The model is the orchestrator, not the calculator. Language understanding is probabilistic and flexible. The math is always deterministic.

// 05Guardrails & Prompt Injection Defence

We had two distinct problems, and they needed different solutions. The first was external — users trying to manipulate the system. The second was internal — the model's own behaviour causing runaway chains when given ambiguous inputs.

Figure 5 — Three-layer defence: keyword filter blocks obvious injection; intent classifier rejects off-domain requests; confidence gate prevents ambiguous routing
# Confidence gate — orchestrator must justify routing before proceeding
ORCHESTRATOR_GATE_PROMPT = """
Before calling any agent, output a JSON routing summary:
{
  "understood_intent": "...",       // what you believe the user is asking
  "primary_agent": "...",           // which agent you are calling and why
  "confidence": 0.0-1.0,           // your confidence in this routing
  "ambiguities": []                 // anything unclear in the request
}
If confidence < 0.75 or ambiguities is non-empty:
  DO NOT call any agent.
  Return a clarification request to the user instead.
"""

def validate_routing_summary(summary: dict) -> bool:
    return (
        summary.get("confidence", 0) >= 0.75 and
        len(summary.get("ambiguities", [])) == 0 and
        summary.get("primary_agent") in KNOWN_AGENTS
    )

// 07Migration: Don't Rewrite, Rewrap

The question every team asks: "Do we rewrite everything to go agentic?" Almost always, the answer is no. The legacy code you're tempted to throw away encodes years of regulatory knowledge, edge cases, and validated financial logic. A startup building from scratch doesn't have that. It's your moat.

Figure 7 — The migration pattern: wrap, don't rewrite. Legacy code stays untouched. Clean wrapper + rich schema = reliable tool calls.

Three Things That Bite You During Migration

Parameter naming — rename abbreviations in the wrapper. r → annual_interest_rate_percent. acct_id → account_identifier. LLMs infer from names, not comments.

Side effects — audit every function before exposing it. Anything that writes, logs, or triggers downstream processes must be sandboxed or wrapped with an explicit confirmation step. The model may call it ten times in a loop.

Error handling — your existing exceptions were designed for developers. Convert them to model-friendly errors before they surface:

def safe_tool_wrapper(fn):
    def wrapped(**kwargs):
        try:
            result = fn(**kwargs)
            return {"status": "ok", "result": result}
        except ValidationError as e:
            return {"status": "error", "reason": "invalid_parameter",
                    "detail": str(e), "suggestion": "Check parameter types and retry"}
        except DatabaseError:
            return {"status": "error", "reason": "data_unavailable",
                    "suggestion": "Account data temporarily unavailable — retry in 30s"}
        # Model can reason about this. It cannot reason about a stack trace.
    return wrapped

// 08Intelligent Model Routing

Not every query deserves the same model. A portfolio balance lookup and a 30-year retirement plan are fundamentally different tasks — treating them identically is expensive and slow.

Figure 8 — Intelligent model routing. ~60-70% of queries are simple lookups that don't need your most capable model.
Groq is an inference provider (fast, open-source models — LLaMA, Mixtral). Grok is xAI's model. Know which you mean before you say it on air.
Don't over-engineer the router. A complex routing layer is just another failure point. Keep heuristics simple and err toward the capable model when in doubt. A wrong answer from under-routing is more expensive than a higher API bill.

// 09What I'd Tell My Past Self

If I could go back to the start of this project with everything I know now, it would be this:

Function calling is not a feature. It's a discipline. Most teams treat it as the former — they wire up a few tools, it works in the demo, and they move on. The teams that treat it as a discipline — with schema governance, token budgets, observability, security review, and failure testing built in from day one — those are the ones that actually ship reliable AI systems in production.

The gap between those two groups is where most of the pain in this industry right now is living. And it's entirely closeable — not with better models, but with better engineering around the models you already have.

The Five Things That Actually Matter

Schema quality determines tool reliability. More than the model version, more than the prompt length. Get the parameter names and descriptions right.

Determinism is a design choice, not a model property. Put your math in code. Let the model handle language.

Circuit breakers must live in code. The model cannot reliably stop itself. Your middleware must.

Observability is not optional in production. You cannot debug what you cannot trace. Every hop needs a log, every session needs a trace ID.

Migration is an API design problem. Wrapping existing logic correctly — names, descriptions, error contracts — is the real work. The AI part is the easy part.

If any of these patterns resonate with problems you're solving, I'd love to hear how your team is approaching it. The engineering discipline around function calling is still being written — and the teams building it carefully right now are defining what reliable AI in production looks like.
