// 01What Is Function Calling

Think of a really smart colleague who has deep knowledge and strong reasoning skills, but no direct access to your systems, your database, or the internet. If you ask them "what is the current balance in my retirement account?", they cannot look it up. But they can tell you exactly what needs to be looked up and how. They write a precise request: get me the retirement account balance for this user, as of today. You take that request, call the actual system, bring back the answer, and they use it to reason and respond.

That is function calling. The LLM is that colleague. It cannot directly touch your database, your APIs, or your financial systems. What it can do is emit structured, precise requests describing exactly what it needs. Your application layer intercepts those requests, executes the actual calls against real systems, and returns the results. The model folds those results into its reasoning and produces a response.

In practice, these function calls are defined using standards and protocols that tell the AI what tools are available, what each tool does, and how to call it. One example is MCP, the Model Context Protocol, an open standard for connecting AI models to external tools and data sources in a consistent way. Much like how USB is a standard that lets you plug any device into any computer without worrying about compatibility, MCP means the AI does not need to know how a tool works internally. It just needs to know the tool exists, what it expects as input, and what it returns.

An agent is what you get when you give that model a goal and let it decide which function calls to make, in what order, and based on what it gets back. A multi-agent system is several specialised agents working under a common orchestrator, each responsible for its own domain and its own set of tools.

The model is not calling functions. It is emitting structured text that describes what it wants called. Your application decides what happens next.

Here is a concrete example. When you ask an AI assistant "what is the weather in New York?", the model does not know the weather. Internally, it emits something like:

{
  "type": "tool_use",
  "name": "get_weather",
  "input": {
    "location": "New York",
    "units": "fahrenheit"
  }
}

Your application layer intercepts that, calls the actual weather API, gets the result, and passes it back. The model then writes the response the user sees. The user sees none of this. But a full request-response loop just completed, and that loop is the fundamental primitive of everything that follows.

The model is the reasoning engine. Function calling is how it acts on the world.

// 02The Financial Planning Architecture

Let me make all of this concrete with a single question. A user opens their AI financial assistant and types: "Am I on track to retire at 62?"

That is seven words. But for the system to answer it meaningfully, a lot has to happen underneath. The LLM reads the question, understands the intent, and realises it cannot answer from memory alone. It needs real data. So it starts making function calls. One to the portfolio service asking for the user's current savings and investment balance. Another to the tax engine asking for a projected liability based on planned withdrawal amounts. Another to a Monte Carlo simulation model that runs thousands of scenarios to return a probability range rather than a single number, because a single number here would be false precision. And finally, a call to a regulatory rules engine that checks a genuinely complex set of constraints: IRS 401(k) annual contribution limits of $23,500 in 2025 with a $7,500 catch-up for those over 50, Required Minimum Distribution rules requiring mandatory withdrawals starting at age 73 or 75 for those born after 1960 under SECURE 2.0, the 10% early withdrawal penalty before age 59 and a half, Social Security claiming strategy between ages 62 and 70, and applicable ERISA obligations for any employer-sponsored plans involved.

In our architecture, each of those function calls is made through a defined tool interface. The agent knows what tools exist, what each one expects as input, and what it returns. That contract between the agent and the tool is what protocols like MCP formalise, so the agent does not need to know how the portfolio service is built internally.

This is also where the multi-agent design becomes necessary. The orchestrator agent receives the user's question and routes it. A retirement specialist agent handles the projection logic. A tax specialist agent handles the liability estimate. Each makes its own function calls to its own set of tools. The orchestrator brings the results together into a coherent answer.

Figure 2 Multi-agent financial planning architecture. One user question cascades through orchestrator → specialist agents → tool layer → deterministic code.

Notice what is happening at each layer. The LLM orchestrator handles language understanding and routing. Specialist agents handle domain-specific coordination. The tool layer wraps deterministic functions. The deterministic code layer is pure math: same input, same output, every time.

Four or five function calls, across two or three agents, each waiting on the previous, each a potential failure point. What if the tax estimator times out on the third call? What if the Monte Carlo returns a range of 71% to 84% probability of success and the model has to reason about uncertainty rather than a clean answer? What if an input somewhere in that chain tries to redirect which account gets queried?

One seven-word question exposes every hard problem in function calling simultaneously.

What if the tax estimator times out on call three? What if the Monte Carlo returns a range and the model has to reason about uncertainty? What if an input tries to redirect which account gets queried? One question, every hard problem.
Each hop in this chain is a potential failure point. What if the tax service times out on call three? What if the Monte Carlo model returns a probability range and the model doesn't know how to reason about uncertainty? What if a user input redirects which account gets queried?

// 03The Deadlock Incident

Six months ago, building this system on AWS Bedrock, we experienced every engineer's nightmare: an infinite loop that burned our entire token budget before we knew what happened.

We built a financial planning proof of concept using a multi-agent architecture, with an orchestrator agent sitting above a set of specialist agents handling retirement planning, education planning, and personal wealth management. We wrote the orchestrator prompt to say, roughly: "here are your agents, decide who to call." And we moved on.

The first time we ran a complex query, something like "build me a comprehensive plan covering my retirement and my kid's college," the system hung. No response. We waited a few minutes. Nothing.

What had happened: the orchestrator called the retirement agent. The retirement agent, because its prompt was not constrained tightly enough, decided it needed broader context and called back up to the orchestrator. The orchestrator treated it as a new request and called the retirement agent again. Agent A calling Agent B calling Agent A.

We did not catch it from a log alarm or a timeout. We caught it when we started getting an Out of Tokens error. Our POC was provisioned with a 1 million token budget, and the loop had burned through all of it before we knew anything was wrong. We only found out by starting to debug the error.

Figure 3 The deadlock: Orchestrator calls Retirement Agent; Retirement Agent calls Orchestrator; repeat until token budget is exhausted

What We Fixed

Three independent defences, each catching what the others miss.

First, a hard token ceiling per agent, 4,000 tokens maximum, enforced in middleware, not the prompt. Second, a call depth counter that kills the session if the same agent is invoked more than a set number of times within 60 seconds. Third, the orchestrator prompt was rewritten to explicitly forbid upward delegation. It now says: you only call tools, you do not call other agents.

The lesson: you cannot rely on the model to know when to stop. That logic has to live in your code.

# Token budget enforced in middleware, not the prompt
class AgentBudgetMiddleware:
    MAX_TOKENS_PER_AGENT = 4000

    def before_invoke(self, session_id, agent_name, token_count):
        running = self.tracker.get(session_id, agent_name)
        if running > self.MAX_TOKENS_PER_AGENT * 0.9:
            raise GracefulBudgetExit(
                agent=agent_name,
                message="Budget ceiling reached, returning partial result"
            )

# Call depth counter catches loops before exhaustion
def check_call_depth(session_id, window_seconds=60, max_calls=12):
    recent = call_log.count_since(session_id, window_seconds)
    if recent > max_calls:
        raise LoopDetected(f"Session {session_id}: {recent} calls in {window_seconds}s")

# Orchestrator prompt forbids upward delegation
ORCHESTRATOR_SYSTEM = """
You are a financial planning orchestrator.
STRICT RULES:
- You ONLY call tools. You do NOT call other agents.
- If you cannot route with confidence, return a clarification request to the user.
- Never attempt to resolve ambiguity by calling multiple agents sequentially.
"""

// 04The Hybrid Deterministic Architecture

Early in the project we debated how much to trust the model to compute financial outcomes directly. We ran an experiment: we asked the LLM to calculate a simple 20-year retirement projection from a prompt, provided the same inputs across five runs, and got four different answers. The variance was small in absolute terms, but in financial planning, small differences in projected returns compound into significantly different advice. We would never want to ship that.

That was the decision point. We decided to treat the model and AI agents as orchestrators: good at reasoning and taking actions, but not at calculating. The moment the system needs to compute anything that matters, a projection, a tax estimate, or a risk score, it calls a deterministic tool. Same input, same output, every single time.

The model is the orchestrator, not the calculator. Language understanding is probabilistic and flexible. The math is always deterministic.

Figure 4 The hybrid architecture: LLM handles intent and routing; deterministic code handles computation

In practice, this meant wrapping our existing rule-based calculation logic and manual process workflows as tools. We had been migrating a legacy system, a combination of rule-based engines and paper-based manual processes, to a modern platform. Rather than rebuilding that logic from scratch for the AI layer, we wrapped it. The model calls the tool, the tool runs the existing validated code, and the result comes back. None of the financial logic changed. This not only gives us deterministic results but also lets us reuse business rules that took years to develop.

We also learned that parameter naming matters far more than you would expect. A function parameter named "r" for interest rate, the model will not reliably populate it. Rename it "annual_interest_rate_percent" and it works consistently every time. The model infers meaning from parameter names, not from documentation it cannot see at execution time.

# ❌ This will produce inconsistent tool calls
def calc_ret(sv, r, n, mc=10000):

# ✅ This produces reliable tool calls every time
def calculate_retirement_projection(

// 05Guardrails & Prompt Injection Defence

We had two distinct problems, and they needed different solutions. The first was external: users trying to manipulate the system. The second was internal: the model's own behaviour causing runaway chains when given ambiguous inputs.

Both came out of our internal Dev/QA security testing phase. Before moving toward production, we built a set of AI agents specifically designed to attack the system, using LLMs as testers, or what is sometimes called LLM as a Judge. Rather than writing static adversarial test cases by hand, these agents generated prompts dynamically and evaluated whether the financial planning agent could be manipulated. The failure modes they found were more creative and varied than anything we would have written manually.

We fixed these at three layers. The first is a keyword filter at the input boundary, a fast pattern-based check that flags two distinct categories: known prompt injection signals such as phrases like "ignore your previous instructions," "forget your system prompt," "act as if," or "override," and terms clearly outside the financial planning domain such as account transfer instructions, references to other clients' data, or anything that looks like privilege escalation. If either category triggers, the request is rejected immediately before it touches the model. The second layer is an intent classifier, a lightweight model that reads the query and asks whether this is a legitimate financial planning request, requiring a confidence score above 0.85 to proceed. The keyword filter catches the obvious cases fast; the classifier catches semantic equivalents that rephrase the same attack. The third layer is at the orchestrator level: a confidence gate that requires the orchestrator to emit a structured routing summary before calling any agent. If confidence drops below 0.75 or there are unresolved ambiguities, it stops and asks the user to clarify rather than proceeding into an uncertain chain.

Figure 5 Three-layer defence: keyword filter blocks obvious injection; intent classifier rejects off-domain requests; confidence gate prevents ambiguous routing
# Confidence gate orchestrator must justify routing before proceeding
ORCHESTRATOR_GATE_PROMPT = """
Before calling any agent, output a JSON routing summary:
{
  "understood_intent": "...",       // what you believe the user is asking
  "primary_agent": "...",           // which agent you are calling and why
  "confidence": 0.0-1.0,           // your confidence in this routing
  "ambiguities": []                 // anything unclear in the request
}
If confidence < 0.75 or ambiguities is non-empty:
  DO NOT call any agent.
  Return a clarification request to the user instead.
"""

def validate_routing_summary(summary: dict) -> bool:
    return (
        summary.get("confidence", 0) >= 0.75 and
        len(summary.get("ambiguities", [])) == 0 and
        summary.get("primary_agent") in KNOWN_AGENTS
    )

// 06Observability & Preventive Monitoring

After the deadlock, our philosophy shifted entirely, from reactive to preventive. Not "alert me when it breaks," but "alert me before it can break." And to do that properly in an agentic system, you have to rethink what observability even means. In a traditional application you monitor hosts: CPU, memory, uptime. In an agentic AI system, none of that tells you anything useful. What you monitor instead is behaviour, cost, and correctness.

The first thing we instrumented was token usage. In our setup, token counts per model invocation flow into a metrics system, and we set per-session rolling alarms on them, not on aggregate daily spend, but on what a single session is consuming in real time. If any session climbs past 3,600 tokens on what should be a lightweight agent interaction, the alarm fires before we hit our 4,000 token ceiling. This is also your earliest loop detector. Token count climbing steeply within a single session is the first signal something is recursing. We did not have this alarm during the deadlock. We only found out by hitting the Out of Tokens error and debugging backward. That alarm would have caught it within the first few iterations.

The second thing we tracked was agent behaviour, specifically tool selection rate and invalid tool invocations. An agent calling the wrong tool, or calling a tool with malformed parameters, tells you that your schema descriptions are drifting from what the model expects, or that an adversarial input is trying to redirect tool behaviour. Agent trace logs record every step in the reasoning chain, every tool call, every intermediate decision, every handoff. Paired with distributed tracing, a multi-agent failure that used to mean correlating timestamps across four different log groups becomes a single trace ID.

For loop detection specifically, we wrote a log query that counts how many times the same session ID appears within a 60-second window. More than eight appearances triggers the alarm. It catches the circular call pattern by behaviour, before token exhaustion, which is the only way to catch it early enough to matter.

On the compliance side, API call logging with full identity context, which agent called which tool, under which role, when, is non-negotiable in a regulated financial environment. Every tool invocation is potentially a compliance event. If a client query later comes under scrutiny, you need to reconstruct the exact chain. That audit trail is not optional.

The last thing we set up was cost anomaly detection. The system learns your normal spending pattern over the past seven days, and if any single day spikes significantly above that baseline, it sends you an alert. We set our threshold at 150%, so if our usual daily spend is $100 and it suddenly hits $150 or more, we get notified immediately. It takes about five minutes to configure and runs automatically after that. Had we set this up before the deadlock, we would have known something was wrong within the first few minutes instead of finding out after the entire token budget was gone.

The most important layer though is not a monitoring service at all. It is budget fencing in middleware. Every agent has a hard token ceiling enforced in code, not in the prompt, not by the model, in code. If a session approaches the ceiling, the agent exits gracefully and returns what it has. Monitoring tells you something went wrong. Budget fencing stops it before it does.

Figure 6 Four-layer observability stack with budget fencing as the hard stop

// 07Migration: Don't Rewrite, Rewrap

The question every engineering team asks when considering this kind of system is: do we really need to rewrite, or can we simply rewrap and save significant time, money, and risk? We were migrating from a combination of rule-based engines and manual paper-based workflows, years of accumulated business logic, regulatory rules, and institutional knowledge encoded in systems that were never designed for AI. The answer we landed on: don't rewrite, rewrap.

What we essentially built is what software architects call an Anti-Corruption Layer, a well-established pattern in microservices design where you place a translation layer between a new system and a legacy one, so neither has to speak the other's language. The legacy side has abbreviated parameter names, raw exceptions, and side effects baked in. The agent layer expects clean tool names, descriptive parameters, and structured responses. The wrapper is what stands between them and keeps both worlds intact.

Figure 7 The migration pattern: wrap, don't rewrite. Legacy code stays untouched. Clean wrapper + rich schema = reliable tool calls.

The Pattern: Wrap and Expose via AgentCore

You take your existing rule-based functions and SQL queries and wrap them as tools, exposed through AWS AgentCore the managed runtime that lets you deploy and scale the agent layer without rebuilding the underlying calculation infrastructure. The model calls the tool; the tool runs the existing validated code; the result comes back. None of the financial logic changed.

Three Things That Turned Out to Be Harder Than They Look

Parameter naming. Abbreviations and short names that developers understand don't translate reliably to model behaviour, and the wrapper is where you fix that. Rename them to be self-describing.

Side effects. Any existing function that writes, logs, or triggers downstream processes has to be sandboxed or wrapped with an explicit confirmation step, because the model may invoke it unpredictably and repeatedly.

Error handling. Existing code throws exceptions designed to be caught by application logic. A model receiving a raw database error simply does not know what to do with it. Every tool wrapper needs to convert failures into structured, model-readable responses that include a reason and a suggested next step.

The migration took longer than the AI work. That was the surprise, and the lesson.

def safe_tool_wrapper(fn):
    def wrapped(**kwargs):
        try:
            result = fn(**kwargs)
            return {"status": "ok", "result": result}
        except ValidationError as e:
            return {"status": "error", "reason": "invalid_parameter",
                    "detail": str(e), "suggestion": "Check parameter types and retry"}
        except DatabaseError:
            return {"status": "error", "reason": "data_unavailable",
                    "suggestion": "Account data temporarily unavailable retry in 30s"}
        # Model can reason about this. It cannot reason about a stack trace.
    return wrapped

// 09Confidence-Gated Execution & Human-in-Governance

Model routing decides which model handles a request. But there's a second routing decision that matters just as much in financial services: based on how confident the system is in its output, what happens next and who is responsible for it?

A common framing in AI is human in the loop. But at the pace of financial calculations a system processing thousands of planning queries across an advisor platform humans cannot be inline approvers of every decision. That's not a practical model, and it's not what regulators actually require. What we've moved toward instead is human in governance. Humans don't approve each calculation. They define the rules, set the thresholds, review the exceptions, and continuously calibrate the system. The governance is upstream, not inline.

Human-in-the-loop assumes humans can keep pace with the system. Human-in-governance acknowledges they can't and designs the oversight layer accordingly.

After the agent chain completes and produces a result, the model scores its own confidence in that output. That score determines the execution pathway.

Figure 9 Confidence-gated execution. Three pathways by confidence score. Human governance sits across all three upstream, not inline.

Below 80% confidence, the system routes the request to a human reviewer. An advisor looks at it, validates it, and decides whether and how to act. This usually happens when the query is ambiguous, the data is incomplete, or the regulatory constraints involved are complex enough that the model flagged genuine uncertainty.

Between 80 and 95%, the system proceeds but with additional validation. It reruns the most critical tool calls, cross-checks the output against boundary conditions, and in some cases runs a second model pass with a stricter prompt to challenge its own answer, or routes to a more capable reasoning model. Think of it as the system asking itself: are you sure? Only if that validation holds does the result go forward.

Above 95%, the system acts autonomously. The calculation completes, the result is returned, and the interaction is logged for audit. No human is inline, but everything is reviewable after the fact, and that is where governance actually lives.

Why the Threshold Is a Governance Decision

The 80 and 95 percent thresholds are not technical defaults. They were set by our risk and compliance teams, documented, and reviewed. In financial services, you cannot say "the AI decided." You need to demonstrate that human judgment was applied at the right points in the process, and that the thresholds defining those points were themselves a deliberate human decision. Confidence-gated execution is how you operationalise that.

The other value of this pattern is that every result falling into the 80 to 95 percent band is a signal. Over time, you can feed those cases back into your reinforcement learning pipeline, using them to tighten prompts, improve tool schemas, and identify data gaps. As the system learns from those edge cases, you watch the confidence distribution shift upward. The score stops being just a safety gate and becomes a leading indicator of system maturity.

# Confidence-gated execution router
def route_by_confidence(result: dict, score: float) -> dict:
    if score < 0.80:
        return human_review_queue.enqueue(result, reason="low_confidence")
    elif score < 0.95:
        validated = rerun_critical_tools(result)
        second_pass = model.verify(validated, prompt=STRICT_VERIFICATION_PROMPT)
        return second_pass if second_pass.confidence >= 0.95 \
               else human_review_queue.enqueue(second_pass, reason="validation_failed")
    else:
        audit_log.record(result, score=score, pathway="autonomous")
        return result  # governance is in the log, not the approval

// 10What I'd Tell My Past Self

If I could go back to the start of this project with everything I know now, it would be this:

Function calling is not a feature. It's a discipline. Most teams treat it as the former they wire up a few tools, it works in the demo, and they move on. The teams that treat it as a discipline with schema governance, token budgets, observability, security review, and failure testing built in from day one those are the ones that actually ship reliable AI systems in production.

The gap between those two groups is where most of the pain in this industry right now is living. And it's entirely closeable not with better models, but with better engineering around the models you already have.

The Six Things That Actually Matter

Schema quality determines tool reliability. More than the model version, more than the prompt length. Get the parameter names and descriptions right.

Determinism is a design choice, not a model property. Put your math in code. Let the model handle language.

Circuit breakers must live in code. The model cannot reliably stop itself. Your middleware must.

Observability is not optional in production. You cannot debug what you cannot trace. Every hop needs a log, every session needs a trace ID.

Migration is an API design problem. Wrapping existing logic correctly names, descriptions, error contracts is the real work. The AI part is the easy part.

Governance is upstream, not inline. At production scale, humans cannot approve every decision. Design your confidence thresholds deliberately, log everything, and review the exceptions. That's the only oversight model that's sustainable in a regulated environment.

If any of these patterns resonate with problems you're solving, I'd love to hear how your team is approaching it. The engineering discipline around function calling is still being written and the teams building it carefully right now are defining what reliable AI in production looks like.
